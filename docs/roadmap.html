Data Sources · Source System 1
Google Calendar Input & Geocoding
Fetch the user's next upcoming calendar event with a location via the Google Calendar API (OAuth2). Geocode the location into GPS coordinates using OneMap. This is the pipeline's entry point — it drives all downstream routing, bus, and weather lookups.
Google Calendar API OAuth2 OneMap Search API google-api-python-client 
# scripts/ingest.py — fetch next real event from Google Calendar
from google.oauth2.credentials import Credentials
from google_auth_oauthlib.flow import InstalledAppFlow
from googleapiclient.discovery import build

SCOPES = ["https://www.googleapis.com/auth/calendar.readonly"]

def fetch_next_calendar_event(con, token):
    creds = Credentials.from_authorized_user_file("token.json", SCOPES)
    service = build("calendar", "v3", credentials=creds)

    events = service.events().list(
        calendarId="primary",
        timeMin=datetime.now(timezone.utc).isoformat(),
        maxResults=10, singleEvents=True, orderBy="startTime"
    ).execute().get("items", [])

    for event in events:
        location  = event.get("location", "").strip()
        start_dt  = event.get("start", {}).get("dateTime")
        if not location or not start_dt:
            continue  # skip all-day / locationless events
        lat, lng = geocode(location, token)
        event_id = f"GCAL_{event['id']}"
        con.execute("""INSERT OR REPLACE INTO calendar_events
            (event_id, title, start_time, location_raw, dest_lat, dest_lng)
            VALUES (?, ?, ?, ?, ?, ?)""",
            [event_id, event.get("summary", "Untitled"), start_dt, location, lat, lng])
        return event_id, lat, lng

    raise ValueError("No upcoming events with a Singapore location found")

# scripts/ingest.py — Flatten OneMap route response
def fetch_routes(origin_lat, origin_lng, dest_lat, dest_lng, token, event_id):
    r = requests.get(
        "https://www.onemap.gov.sg/api/public/routingsvc/route",
        params={"token": token, "start": f"{origin_lat},{origin_lng}",
                "end": f"{dest_lat},{dest_lng}", "routeType": "pt",
                "mode": "TRANSIT", "numItineraries": 3}
    ).json()
    rows = []
    for i, it in enumerate(r.get("plan", {}).get("itineraries", [])):
        rows.append({
            "option_id": f"{event_id}_opt{i}", "event_id": event_id,
            "total_duration_min": it["duration"] // 60,
            "walk_distance_m": it["walkDistance"],
            "num_transfers": it["transfers"],
            "fare": it.get("fare", 0),
        })
    return pd.DataFrame(rows)

Data Transformation
SQL Cleaning, Recommendation Logic & Walk Alternative
Join routes, weather, and alerts into a ranked recommendation. Calculate leave-by time, flag rainy walks, detect MRT disruptions, and rank routes using window functions. The recommendation_reason field has 9 dynamic labels (e.g. "✓ Fastest + direct", "✓ Fewest transfers (N)", "✓ Least walking (Xm)", "⚠ Rain — Xm exposed walk") computed via MIN() OVER (PARTITION BY event_id) window functions — not hardcoded. Weather and disruption warnings appear under "Why chosen:" for the recommended route. Disruption status is filtered to the actual rail modes in the route (MRT-only route shows MRT disruptions only, not MRT/LRT). Output includes: inline first-transit live arrival after the step-by-step legs (X1+X2 for bus; headway estimate ~3-5 min for MRT, ~5-10 min for LRT — no public real-time MRT/LRT arrival API in Singapore), an alt routes section labelled "Other route options (sorted by arrival time)" with [2]/[3] labels (always exactly 2 alternatives — OneMap API hard-caps at 3 total itineraries), consecutive MRT+LRT legs grouped in compact display, per-alt first-transit X1 + disruption note, and a walk alternative with Zone 1/2 health stats plus optional Garmin/Whoop fitness context. Note: alt routes query route_options directly — querying v_enriched_routes WHERE route_rank > 1 returns 47× duplicates because the view cross-joins 47 weather areas.

-- scripts/transform.sql — Core recommendation view

CREATE OR REPLACE VIEW v_enriched_routes AS
SELECT
    r.option_id, r.event_id, r.total_duration_min,
    r.walk_distance_m, r.num_transfers, r.fare,
    e.start_time, e.title, e.dest_lat, e.dest_lng,

    -- When to leave (10 min buffer)
    e.start_time - INTERVAL (r.total_duration_min + 10) MINUTE AS leave_by,

    w.forecast AS weather_forecast,
    w.is_rainy,
    CASE WHEN ta.alert_id IS NOT NULL THEN ta.message ELSE NULL END AS alert_msg,

    -- Recommendation label
    CASE
        WHEN w.is_rainy AND r.walk_distance_m > 400
            THEN '⚠ Rainy — take covered transport'
        WHEN ta.alert_id IS NOT NULL
            THEN '⚠ MRT disruption — add 20 min buffer'
        WHEN r.total_duration_min = MIN(r.total_duration_min)
                OVER (PARTITION BY r.event_id)
            THEN '✓ Fastest option'
        ELSE 'Alternative route'
    END AS recommendation_reason,

    -- Rank routes per event (1 = best)
    ROW_NUMBER() OVER (
        PARTITION BY r.event_id
        ORDER BY
            CASE WHEN w.is_rainy AND r.walk_distance_m > 400 THEN 1 ELSE 0 END,
            r.total_duration_min
    ) AS route_rank

FROM route_options r
JOIN calendar_events e ON r.event_id = e.event_id
LEFT JOIN weather_forecast w
    ON w.fetched_at = (SELECT MAX(fetched_at) FROM weather_forecast)
LEFT JOIN train_alerts ta
    ON ta.severity = 'HEAVY'
   AND ta.fetched_at > NOW() - INTERVAL '30 minutes';

"The transformation layer joins three data streams. Leave-by time is start_time minus total duration minus a 10-minute buffer — a documented business rule, not a magic number. A window function ranks routes per event: if it's raining and a route has more than 400 metres of walking, that route is penalised and drops in the ranking. The recommendation reason is dynamic — the SQL view produces 9 distinct labels using MIN() OVER (PARTITION BY event_id) window functions, so the label reflects what actually makes each route best: fastest and direct, fewest transfers, least walking, or a rain warning. Weather and disruption notes appear directly under 'Why chosen:' for the recommended route, and disruption is filtered to only the rail lines that route actually uses. After the step-by-step legs, live arrivals show actual clock times — for example, '09:34 (~4 min from now) | if missed → 09:39 (~9 min from now)' — so the user knows exactly when to board, not just a vague wait time. If the recommended route is entirely on foot — for example the destination is just across the street — walk metrics appear inline in the recommendation section: distance, estimated time, step count, calories, and heart-rate zone guidance. The walk alternative section below is only shown when there is an actual transit route above it and the walk is a genuine alternative. Below the recommended route, two alternative routes are listed, sorted by arrival time and labelled [2] and [3] — always exactly two because OneMap caps routing at 3 total itineraries. Each alternative shows compact legs with stop counts, its own first-transit X1 live arrival, and any disruption or delay note for that specific route."

Production Quality · Day 2
Retry, Idempotency & Data Quality
The LTA API times out during rush hour. The OneMap token expires silently. A double-run creates duplicate rows. Add three things to every API call: exponential backoff retry, INSERT OR REPLACE so re-runs are safe, and coordinate validation that rejects bad GPS data before it reaches DuckDB.
retry + backoff INSERT OR REPLACE coord validation pipeline_runs log 
# scripts/ingest.py — wrap every requests.get() in this function
import time, logging

def fetch_with_retry(url, headers=None, params=None, max_retries=3):
    for attempt in range(max_retries):
        try:
            resp = requests.get(url, headers=headers, params=params, timeout=10)
            resp.raise_for_status()
            return resp.json()
        except requests.RequestException as e:
            if attempt == max_retries - 1: raise
            wait = 2 ** attempt  # 1s → 2s → 4s (exponential backoff)
            logging.warning(f"Attempt {attempt+1} failed. Retrying in {wait}s")
            time.sleep(wait)

# Validate GPS is inside Singapore before inserting
def validate_sg_coords(lat, lng):
    if not (1.15 <= lat <= 1.47 and 103.6 <= lng <= 104.1):
        raise ValueError(f"Coords ({lat}, {lng}) outside Singapore bounds")

# Idempotent insert — safe to re-run, no duplicates
con.execute("""
    INSERT OR REPLACE INTO weather_forecast
    SELECT * FROM staging_weather
""")

Why each matters
Retry/backoff — LTA returns 429 or 500 during peak hours. One failure kills the run without retry.
Idempotency — If the pipeline crashes halfway and restarts, INSERT OR REPLACE prevents duplicate rows.
Coord validation — OneMap returns (0, 0) for unresolved addresses. A bus stop in the Indian Ocean corrupts all downstream recommendations.

Raw Data Zone · Day 2
Parquet as Intermediate Storage
Before loading API responses into DuckDB, save them as Parquet files in a date-partitioned raw zone. This gives you replayability: if your SQL has a bug, fix it and re-transform from the raw files without re-calling the APIs. It also creates an audit trail — you can see exactly what LTA returned at any timestamp.
Parquet columnar storage date partitioning raw zone read_parquet 
# scripts/ingest.py — save raw before loading to DuckDB
from pathlib import Path
from datetime import date

def save_raw_parquet(df, source_name):
    today = date.today().isoformat()          # e.g. "2026-07-15"
    path = Path(f"data/raw/{source_name}/date={today}")
    path.mkdir(parents=True, exist_ok=True)
    out = path / f"{source_name}_{today}.parquet"
    df.to_parquet(out, index=False, compression="snappy")
    return out

# Use it in your ingestion flow:

# scripts/api.py — uvicorn scripts.api:app --reload --port 8000
from fastapi import FastAPI, HTTPException
import duckdb

app = FastAPI(title="SG Commute API", version="1.0")

def get_db():
    return duckdb.connect("db/commute.duckdb", read_only=True)

@app.get("/health")
def health():
    return {"status": "ok"}

@app.get("/api/v1/recommendation/{event_id}")
def get_recommendation(event_id: str):
    rows = get_db().execute("""
        SELECT event_id, title, leave_by, total_duration_min,
               fare, weather_forecast, is_rainy, alert_msg,
               recommendation_reason
        FROM v_enriched_routes
        WHERE event_id = ? AND route_rank = 1
    """, [event_id]).df().to_dict(orient="records")
    if not rows: raise HTTPException(404, "Event not found")
    return rows[0]

@app.get("/api/v1/pipeline/status")
def pipeline_status():
    return get_db().execute("""
        SELECT source, status, rows_upserted, ran_at
        FROM pipeline_runs ORDER BY ran_at DESC LIMIT 10
    """).df().to_dict(orient="records")

Orchestration · Day 2
Airflow Pipeline DAG

from datetime import datetime, timedelta
from airflow.operators.bash import BashOperator

PROJECT_DIR = str(Path(__file__).parent.parent)

def _is_evaluation_window(**ctx):
    return datetime.now(timezone.utc).astimezone(SGT).hour == 8

with DAG(
    dag_id="commute_pipeline",
    schedule="*/30 * * * *",       # every 30 minutes

    catchup=False,

Data Modelling, Lake vs Warehouse & dbt
Day 3 covers the conceptual layer beneath your pipeline: how to model data, where to store it, and how industry uses dbt to bring structure to SQL transforms. Your project already implements all of these patterns — this station shows you how to name and explain them confidently in your video.
Star Schema Data Lake Data Warehouse dbt concepts SCD Type 1 Data Lineage
Your project = Data Lake + Data Warehouse
Data Lake — data/raw/ Parquet files. Raw API responses, never modified, date-partitioned, replayable. No schema enforced at write time.
Data Warehouse — db/commute.duckdb. Cleaned, typed, relational schema. Queried by dashboards and APIs.
Transactional DB (not us) — Postgres / MySQL stores live app state row-by-row. DuckDB is columnar-analytical — designed for reads and aggregations, not write-heavy app workloads.
Your DuckDB schema = Star Schema
Fact table — route_options: each row is one commute option. Has measurable facts: duration_min, fare, walk_distance_m.
Dimension tables — weather_forecast, bus_arrivals, train_alerts: descriptive context that joins to routes via time or service_no.
SCD Type 1 — INSERT OR REPLACE overwrites old rows. The 8am weather reading is replaced by the 10am one. No history kept. Type 1 = latest value only.
dbt vs our transform.py — same goal, different tool
Our approach (transform.py)          dbt equivalent
────────────────────────────         ──────────────────────────────────────
Raw Python + raw SQL strings         models/staging/stg_routes.sql
Manually run as a script             dbt run  (compiles + executes all models)
No built-in data tests               dbt test (schema.yml: not_null, unique)
No lineage tracking                  dbt docs generate → visual lineage graph
One file for all logic               staging → intermediate → mart layers
INSERT OR REPLACE written by hand    materialization: table / incremental
Both produce the same output: clean joined data ready for serving. dbt adds governance and structure at the cost of a new tool. For a project this size, transform.py is a valid and common production choice.
Why NOT Kafka / Flink here
Kafka is for data that changes every second — click streams, payments, IoT sensors. Bus arrival data refreshes every 2 minutes; weather every 2 hours. A 30-minute Airflow batch schedule is perfectly adequate.
Flink processes Kafka streams with stateful windowing. No Kafka stream → no Flink. Airflow covers our orchestration needs entirely.
Why NOT Spark / PySpark here
Spark is designed for TB-scale data across a distributed cluster. Our dataset: ~50 route rows per run, ~5,000 bus stops, ~47 weather areas. Spark's startup and shuffle overhead would dwarf actual processing time.
DuckDB handles GB-scale analytical queries on a single machine in milliseconds. It is the correct tool choice for a pipeline of this size.

Shadow → Canary → Promote
Shadow — new model sees same live inputs, predictions logged only, never shown to users. Zero risk.
Canary — gradually shift real traffic: 95%/5% → 50%/50% → 100%/0%. Pause if error rate or latency spikes.

Three Drift Types (Day 4)
Data Drift — input feature distributions change. Example: user changes jobs, hour_of_day shifts from 8 AM to 10 AM starts.
Concept Drift — feature-to-target relationship changes. Example: new MRT line opens; same hour_of_day now predicts a shorter total_duration_min.
Training-Serving Skew — feature computed differently in training vs production. Example: is_rainy encoded as True/False at training time but 1/0 in production — model was never trained on these values.

RAPIDS ecosystem — when you'd actually use it
cuDF — GPU DataFrame (pandas API). Use when data is GB-scale and pandas becomes the bottleneck. Not our scenario: we process ~50 rows per run.
cuML — GPU-accelerated ML (scikit-learn API). Use when model training takes hours due to dataset size. Our Random Forest trains in under 1 second on CPU.
Dask-cuDF — distributed GPU DataFrames across a cluster. Use for TB-scale ETL. Overkill for 47 weather rows.
CuPy — NumPy replacement on GPU. Use when numerical computation is the bottleneck. Our bottleneck is API latency, not computation.
Triton vs FastAPI — what the choice actually means
FastAPI — Python ASGI server. Calls model.predict() directly. Startup in 1s. No GPU required. Correct for <100 predictions/min.
Triton — NVIDIA's model serving runtime. Handles TensorRT/ONNX/TorchScript. Dynamic batching across concurrent requests. Production-scale at >1000 req/s.
Our FastAPI /api/v1/prediction/{event_id} scores 3 route options in under 1ms. Triton's deployment overhead (GPU server, model repository, gRPC client) would cost more than the rest of the pipeline combined. FastAPI is correct.
Metabase vs Streamlit
Metabase — no-code BI builder for non-technical stakeholders. SQL questions → charts. Requires its own Docker service + PostgreSQL metadata store.
Streamlit — Python-code dashboard. Full control over layout, ML panels, and 60s live refresh. No extra SQL layer. Correct for a developer-owned single-user project. Metabase adds infrastructure for zero benefit here.
Feature Store — what it solves (and why we don't need one)
A Feature Store centralises engineered features so multiple teams and models share consistent definitions — preventing training-serving skew at scale.
We have one model and one engineer. Features are computed inline in the SQL training query. A Feature Store (e.g. Feast, Hopsworks) would add a registry, a serving layer, and a separate service — to avoid a problem we don't have. Would become necessary if the project grew to 10+ models across multiple teams.

  • SQL Transformation Designed a SQL transformation layer joining route options, weather forecasts, and disruption alerts; implemented window-function-based route ranking with business rules (rain penalty, disruption buffer) to generate per-event commute recommendations.
  • Production Robustness Implemented exponential backoff retry on all API calls, idempotent upsert pattern (INSERT OR REPLACE), GPS coordinate validation, and a pipeline_runs monitoring table — making the pipeline safe to run in production.
  • Data Serving API Designed a REST API serving layer with FastAPI exposing commute recommendations as a queryable endpoint with auto-generated Swagger documentation — decoupling the recommendation engine from any specific frontend.
  • ML Models Built scikit-learn Random Forest models for commute duration and crowd prediction, scoring all route options per event using engineered weather, route, and bus-arrival timing features; predictions are stored per route, backfilled with observed/proxy actuals, and evaluated with 7-day MAE.
  • Orchestration & Infrastructure Orchestrated the pipeline with an Apache Airflow DAG (7 tasks, 30-minute schedule, gate_evaluate ShortCircuitOperator for daily 8 AM ML evaluation); containerised the full stack with Docker Compose — pipeline, API, and dashboard as three services with volume-mounted persistent storage. The Docker pipeline service uses a custom scheduler.py with a 7-state adaptive machine (NO_EVENT → EVENT_DETECTED_BURST → WATCHING → LEAVE_WINDOW → IN_TRANSIT → ARRIVAL_VERIFY → POST_ARRIVAL_COOLDOWN) instead of a fixed sleep loop.
  • Data Architecture Implemented a lake-warehouse pattern: date-partitioned Parquet raw zone (data lake) feeds a typed DuckDB analytical store (data warehouse). Schema follows star schema design with route_options as fact table and weather, bus arrivals, and train alerts as dimension tables. INSERT OR REPLACE implements SCD Type 1 for all dimensions.
