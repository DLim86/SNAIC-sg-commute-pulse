Hi, I'm a Data Engineering student at SIT. This is my Week 2 assessment — SNAIC SG Commute Pulse.

The problem: I look at my Google Calendar, see a meeting at 10am somewhere in Singapore, and I have no idea which bus or MRT to take, whether there are disruptions, or if I need to leave earlier because of rain. I'm checking three different apps to answer one question: when do I leave?

System Architecture — 3 Layers
1:30 – 3:00
Screen Action
  • Open docs/roadmap.html — show the 3-phase pipeline diagram at the top
  • Point at each phase as you name it
Say This

The system has three layers, each mapped to a rubric criterion.

Layer 1 — Batch pipeline. An Airflow DAG runs every 10 minutes. It fetches four data sources, cleans and stores data in DuckDB, and runs a SQL transformation. This is the core data engineering pipeline.

Layer 2 — ML predictions. After each transformation, a Random Forest model predicts the commute duration for the next event and compares it with what actually happened. A daily Airflow task evaluates 7-day prediction accuracy. This satisfies the ML and real-time output criterion.

Layer 3 — Live serving. A Streamlit dashboard auto-refreshes every 5 minutes showing the current recommendation, the ML prediction, and weather and disruption warnings. A FastAPI layer exposes the same data as a REST endpoint so any client can consume it.

My architecture follows the lake-warehouse pattern. Raw API responses land in date-partitioned Parquet files — that's the data lake. Cleaned, typed data lives in DuckDB — that's the data warehouse, modelled as a star schema with route_options as the fact table.

Data Sources — Where the Data Comes From
3:00 – 4:30
Screen Action
  • Open config_example.py in VS Code — show the placeholder structure, never real keys

Say This

I have five data sources — all real, all live.

Google Calendar via OAuth2. This is the pipeline's entry point. I use the Calendar API to scan my next 10 upcoming events and find the first one that has a location field. OAuth2 handles authentication — on first run it opens a browser for consent, then auto-refreshes the token. The location string is geocoded into GPS coordinates by OneMap. If no calendar event is found, a time-of-day heuristic kicks in: 8 to 10 AM routes to my configured school address, after 4 PM routes home, and after 6 PM it checks my IP location — if I'm already within 3 km of home, the pipeline skips cleanly.

OneMap Singapore. Two endpoints: geocoding — turning an address into GPS coordinates — and public transport routing, which gives me up to 3 route options with duration, walking distance, transfers, and estimated fare. The geocoder uses a progressive fallback: it tries the full calendar location, then strips ", Singapore", then just the first part of the address — because users type locations in many formats. The OneMap JWT token expires every 3 days — the pipeline always calls for a fresh one at startup, never caches it to disk.

LTA DataMall. Two endpoints: real-time bus arrivals by stop code — next arrival time and how crowded the bus is — and active MRT disruption alerts with severity levels. I registered for this API and got approval. The key lives in config.py which is gitignored — it never goes to GitHub.

data.gov.sg weather. Open API, no registration needed. Returns a 2-hour forecast broken down by area across Singapore. I flag any area with "rain" or "shower" in the forecast as is_rainy = TRUE at ingest time.

ip-api.com. Free geolocation API. Used to detect where I currently am — my routing origin — without needing GPS access from a Python script.

If Asked About Security

All API keys live in config.py which is in .gitignore. The template is config_example.py — only placeholders, committed to the repo. Running git status confirms config.py doesn't appear as a tracked file. Google's OAuth token and credentials files are also gitignored.

Batch Pipeline — What We Did to the Data
4:30 – 6:30
Screen Action
  • Open Airflow UI at http://localhost:8080 — show the commute_pipeline DAG
  • Show the task graph: calendar → geocode → [weather, bus, alerts] → transform → predict
  • Click on one completed task run — show the green success state and click Logs
  • Briefly show a Parquet file in data/raw/ in VS Code Explorer
Say This

The Airflow DAG runs every 10 minutes on a cron schedule — */10 * * * *. It has 7 tasks. The first two run sequentially: fetch the calendar event, then geocode the destination. The next three run in parallel — fetch weather, bus arrivals, and train alerts simultaneously because they don't depend on each other. Then SQL transformation runs once all three complete. Finally, the prediction task scores the next event against the ML model.

I used Airflow instead of a cron job because it gives me this UI — I can see every task run, whether it succeeded or failed, and click into the logs to debug. A cron job has no visibility.

Now let me show what the pipeline actually does to the data. Every API call goes through a retry wrapper — 3 attempts, exponential backoff of 1 second, 2 seconds, 4 seconds. LTA's API returns errors during rush hour. Without retry, one blip at 8am kills the whole run.

Before loading to DuckDB, raw responses are saved as Parquet files in a date-partitioned folder — you can see them here in data/raw/. That's the data lake layer. The reason is replayability: if I find a bug in my SQL transformation, I can fix it and re-run from the raw files without calling the APIs again.

Every insert uses INSERT OR REPLACE — never a bare INSERT. If the pipeline crashes halfway and restarts, it produces the same result as a clean run. No duplicate rows. That property is called idempotency and it's essential for any scheduled pipeline.

If Asked: What is the star schema?

route_options is the fact table — each row is one commute option, with measurable values: duration, fare, walk distance. weather_forecast, bus_arrivals, and train_alerts are dimension tables that join to routes. route_options is the centre of the star, dimensions radiate outward.

SQL Transformation — Ranking Routes & Leave-By Time
6:30 – 7:30
Screen Action
  • Open scripts/transform.py in VS Code — scroll to the BEST_ROUTE_QUERY at line 40
  • Show the v_enriched_routes view definition in scripts/schema.py
  • Run python scripts/transform.py in terminal — show the output with LEAVE LATEST, LEAVE NOW, step-by-step legs
Say This

The transformation layer is a SQL view called v_enriched_routes. It joins route options with the latest weather and any active MRT alerts. The key logic is in two places.

First: leave-by time. start_time MINUS INTERVAL (duration + 10) MINUTES. The 10-minute buffer accounts for walking from home to the bus stop — that's a documented business rule, not a magic number.

Second: route ranking. I use a window function — ROW_NUMBER() OVER (PARTITION BY event_id ORDER BY penalty, duration). If it's raining and a route has more than 400 metres of walking, that route gets a penalty and drops in the ranking. The fastest route wins unless rain makes it impractical. The dashboard always shows rank 1.

The output shows two departure times: leave latest — the last safe departure to arrive 10 minutes early — and leave now — what happens if you left at this exact second. Step-by-step legs show each walk, bus, and MRT segment with duration and stop names.

Two additional sections appear after the recommended route. First: a live bus arrivals board — every service currently active at the origin stop, with x₁ (minutes to first bus) and x₂ (minutes to second bus) pulled from the bus_arrivals table, plus crowding level (seated / standing / full). Second: an alt routes section listing the top 3 non-recommended alternatives, each scored by next-bus wait time plus total duration so faster routes with bad bus timing rank below slower routes with a bus arriving now. Each leg in the alt routes shows the number of stops, and any bus more than 10 minutes away gets a warning flag. Note: alt routes query route_options directly — using v_enriched_routes WHERE route_rank > 1 would return 47 copies of each alternative because the view cross-joins 47 weather areas.

Say This

This is the ML component. The model answers the question: given today's conditions, how long will this journey actually take?

I use a Random Forest Regressor from scikit-learn. The features are: hour of day, day of week, whether it's raining, walking distance in metres, and number of transfers. The target is total journey duration in minutes. The model is trained on historical route data stored in DuckDB and saved to a .pkl file in the models/ folder.

Each pipeline run calls the predict task — it loads the model, scores the next calendar event, and inserts the prediction into a predictions table in DuckDB alongside the predicted time and model version. When the commute window passes, the pipeline backfills the actual_min column by comparing with the route that was actually taken.

There's a third component — model evaluation. A daily Airflow task runs every morning at 8am. It reads the predictions table, computes the 7-day Mean Absolute Error comparing predicted versus actual duration, and logs it to the pipeline_runs monitoring table. This is the MLOps feedback loop — the model gets evaluated on a schedule, not just at training time.

You can see the predicted versus actual comparison live in the Streamlit dashboard. The 7-day MAE tells me whether the model is getting better or worse as more real data accumulates.

If Asked: Why Random Forest and not Linear Regression?

Random Forest handles non-linear interactions between features — rush hour combined with rain has a different effect than either alone. Linear Regression would require manual feature engineering for that. RF also handles small datasets gracefully, which matters because I'm bootstrapping with synthetic historical data until enough real runs accumulate.

If Asked: Why batch and not real-time streaming (Kafka)?

Bus arrival data refreshes every 2 minutes. Weather updates every 2 hours. Polling every 10 minutes via Airflow is perfectly adequate — there's no event happening faster than that which would require a Kafka stream. Kafka would add infrastructure complexity (brokers, consumer groups, partitions) with no benefit for this data velocity.

Docker Compose packages the entire system into 3 containers. docker compose up — one command — starts everything. Anyone can clone the repo, add their API keys to a .env file, and run the full system. No Python installation, no virtual environment setup. That's the point of containerisation.

The DuckDB file is volume-mounted — ./db:/app/db — so data persists across container restarts. The pipeline, API, and dashboard share this one file as their common data store.

All three services use a single Dockerfile built from the same requirements.txt. Each service overrides the command field — so the same image runs as a pipeline, an API server, or a dashboard depending on which service definition is used.

Reflection — 3 Mandatory Questions
12:00 – 14:00
Rubric requirement — Required, not optional

A short reflection is required as part of the presentation. Answer all 3 questions.

Question 1 — Most Challenging Part

The most challenging part was a silent API deprecation. The LTA DataMall Bus Arrival endpoint changed from BusArrivalv2 to v3/BusArrival in August 2024 — but there was no obvious error message. Every bus stop code returned HTTP 404 with "The requested API was not found", which I initially thought meant the stop code was wrong. I tried multiple known-active stops and they all failed. Only when I checked the official LTA DataMall API PDF v6.8 did I see that the URL itself had changed. The fix was a single line. The lesson: always check the latest API documentation, not just the URL you were given originally.

The second challenge was the OneMap routing response. Each route leg has a route field that sometimes returns a dictionary — with a shortName key for the MRT line — and sometimes returns a plain string. Calling .get("shortName") on a string crashes Python. I had to add an isinstance(route_field, dict) guard — a defensive check that only becomes obvious when you see the raw API data vary between requests.

A third edge case: Parquet promotes integer bus stop code columns to float64 when any NaN rows exist, turning "65721" into "65721.0". LTA's API rejects codes with a decimal point. The fix was stripping the suffix with str(code).split(".")[0].

A fourth challenge: after rescheduling two calendar events, the pipeline kept showing the wrong destination. The root cause was three bugs acting together. First, the routing origin was being geocoded from HOME_ADDRESS — when the event destination was also home, origin equalled destination and OneMap returned a 404 with no route. Fix: origin is always IP geolocation, never HOME_ADDRESS. Second, rescheduling a calendar event does not remove the old version from DuckDB — INSERT OR REPLACE upserts by primary key only, so the old start_time remained in calendar_events and transform picked whichever future event it found first. Fix: _purge_stale_events() deletes all future events that don't match the current active event_id after every ingest. Third, DuckDB's INSERT OR REPLACE on route_options first deletes the existing row, but that delete fails because route_legs still holds a foreign key reference to it. Fix: pre-delete all route_legs for the event before upserting route_options.

Question 2 — Most Proud Of

The part I'm most proud of is the end-to-end live data integration. Real Google Calendar events drive real API calls — every 10 minutes, Airflow ingests live LTA bus arrivals, live MRT alerts, live weather — and produces a genuine, actionable recommendation that is specific to my actual next meeting and current conditions. This is not a demo with mock data; it's a system I use.

The walk alternative feature with Zone 1 and Zone 2 heart rate context is also something I'm proud of, because it makes the pipeline genuinely personal — it's not just the fastest route, it's the right route for your health and your schedule together.

Question 3 — If I Had More Time

Three improvements. First, I'd add Kafka streaming for bus arrivals. Right now the pipeline polls every 10 minutes. With Kafka, LTA could push arrival updates as they happen and the dashboard would update in seconds, not minutes. This would also let me demonstrate Flink for windowed stream processing.

Second, I'd accumulate more historical data for the ML model. The Random Forest performs better with more training examples. With 30 days of real commute data, the prediction accuracy would be meaningfully better than the synthetic bootstrap I'm using today.

Third, a Telegram bot that pushes the recommendation to my phone 30 minutes before each calendar event. Right now I have to open the dashboard to check — a push notification would make it genuinely zero-friction.
