System Architecture — 3 Layers
1:10 → 2:20 · ~150 words
Screen / Do
  • Open docs/roadmap.html in browser
  • Point at each of the 3 phases as you name them
Say

The system has three layers.

Layer 1 — Batch pipeline. An Airflow DAG runs every 30 minutes. It fetches four data sources, stores them in DuckDB, and runs a SQL transformation. This is the core data engineering pipeline.

Layer 2 — ML predictions. A Random Forest model predicts commute duration and bus crowding for the next event. A daily Airflow task evaluates 7-day accuracy against actual outcomes.

Layer 3 — Live serving. A Streamlit dashboard auto-refreshes every 60 seconds. A FastAPI layer exposes the same data as REST endpoints for any external client.

The architecture follows the lake-warehouse pattern. Raw API responses land in date-partitioned Parquet files — the data lake. Cleaned, typed data lives in DuckDB — the data warehouse, with route_options as the fact table of a star schema.

Optional — only if ahead of time

The star schema: route_options is the centre — duration, fare, walk distance. weather_forecast, bus_arrivals, and train_alerts are dimensions that join to it. The data lake is for replayability; the warehouse is for fast SQL querying.

Batch Pipeline — Ingestion, Retry, Parquet, Idempotency
3:30 → 5:05 · ~200 words
Screen / Do

Say

The Airflow DAG runs every 30 minutes — */30 * * * *. Seven tasks in a sequential chain: schema_check, ingest, transform, predict_commute, backfill_actuals, a ShortCircuitOperator called gate_evaluate that only passes at 8 AM Singapore time, and evaluate_model. The gate means ML evaluation runs once a day even though the DAG fires every 30 minutes.

Airflow gives me full task visibility — green successes, red failures, clickable logs. A cron job can't do that.

Every API call goes through a retry wrapper: 3 attempts, exponential backoff of 1, 2, then 4 seconds. LTA's API returns errors during rush hour — without retry, one blip at 8 AM kills the whole run.

Before loading to DuckDB, raw responses are saved as date-partitioned Parquet files under data/raw/. That's the data lake layer. If I find a bug in my SQL transformation, I can fix it and re-run from the raw files without calling the APIs again.

Every insert uses INSERT OR REPLACE — never a bare INSERT. If the pipeline crashes and restarts, it produces the same result. That property is called idempotency and it's essential for any scheduled pipeline.

Optional — only if ahead of time

route_options is the fact table — duration, fare, walk distance. weather_forecast, bus_arrivals, train_alerts are dimension tables. That's the star schema centred on routes.

SQL Transformation — Ranking Routes & Leave-By Time
5:05 → 6:00 · ~115 words
Screen / Do
  • Open scripts/schema.py in VS Code — scroll to the v_enriched_routes VIEW definition
  • Point at the ROW_NUMBER() OVER (PARTITION BY event_id ...) window function
  • Run python scripts/transform.py in terminal — show LEAVE LATEST, LEAVE NOW, step-by-step legs in output
Say

The transformation layer is a SQL view — v_enriched_routes — that joins route options with the latest weather and active train alerts.

Leave-by time: start_time MINUS INTERVAL (duration + 10) MINUTES. The 10-minute buffer accounts for walking to the bus stop — a documented business rule.

Route ranking: ROW_NUMBER() OVER (PARTITION BY event_id ORDER BY penalty, duration). Rain plus more than 400 metres of walking adds a penalty and drops a route in the ranking.

The "Why chosen" reason is dynamic — nine distinct labels produced by MIN() OVER (PARTITION BY event_id) window functions: fastest, direct, fewest transfers, least walking, cheapest fare, rain warnings. Not hardcoded.

Disruption warnings are filtered to the route's actual rail lines — a bus-only route shows no disruption section at all.

Optional — only if ahead of time

Live bus arrivals show actual clock times — "09:34 (~4 min)" — not a relative wait. MRT headways use fixed estimates: ~4 min MRT, ~7 min LRT — Singapore does not publish a public real-time MRT arrival API for developers. Alt routes always show two alternatives: OneMap hard-caps at 3 total itineraries. Alt routes query route_options directly — not the enriched view — because the view's weather join and ranking aren't needed for the alternatives.

The duration model uses 11 features — timing, weather, route properties, and bus arrival context. Training starts with 500 synthetic rows, labeled as synthetic, using known Singapore commute patterns: rush hour plus 15%, rain plus 8%, weekends minus 20%. Question 1 — Most Challenging Part

The most challenging part was a silent API deprecation. LTA's Bus Arrival endpoint changed from BusArrivalv2 to v3/BusArrival in August 2024 — but every stop code returned HTTP 404 with "The requested API was not found". I thought the stop codes were wrong. I tried multiple known-active stops. They all failed. Only when I checked the official LTA DataMall API PDF v6.8 did I see the URL had changed. The fix was a single line. Lesson: always check the latest API documentation, not just the URL you started with.

A second edge case: Parquet promotes bus stop code columns to float64 when any NaN rows exist, turning "65721" into "65721.0". LTA rejects codes with a decimal point. Fix: str(code).split(".")[0].

Question 2 — Most Proud Of

The part I'm most proud of is the live data integration. Real Google Calendar events drive real API calls — live LTA arrivals, live MRT alerts, live weather — producing a genuine, actionable recommendation specific to my actual next meeting. The pipeline runs on live data. The ML model bootstraps with 500 synthetic rows so it works from day one, then improves as real predictions and actuals accumulate. The walk feature with heart-rate zone context is also something personal — it makes the recommendation relevant to health, not just schedule.

Question 3 — If I Had More Time

Three improvements. First, Kafka streaming for bus arrivals — LTA pushes updates every 2 minutes; a stream would cut dashboard latency from 30 minutes to seconds. Second, more historical data — 30 days of real commutes would meaningfully improve model accuracy over the synthetic bootstrap. Third, a Telegram bot that pushes the recommendation 30 minutes before each calendar event — zero friction, no need to open the dashboard.

Optional — only if ahead of time (Q3 extension)

If this scaled to thousands of simultaneous commuters: replace FastAPI with NVIDIA Triton Inference Server for batched ML inference, replace pandas with cuDF from the RAPIDS ecosystem. But the Day 5 Technology Selection Exercise maps "low-traffic prediction API" to FastAPI and "100MB daily data" to pandas — which is exactly what I built. Knowing when not to use a technology is as important as knowing how to use it.

Optional — stale event bug (Q1 extension if time)

After rescheduling calendar events, the pipeline kept showing the wrong destination. Three bugs acting together: routing origin was geocoded from HOME_ADDRESS (fix: always IP geolocation); rescheduling doesn't remove old DB rows (fix: _purge_stale_events()); DuckDB's INSERT OR REPLACE on route_options failed because route_legs held a FK reference (fix: pre-delete route_legs before upserting route_options).

Q&A Reference — Not part of the recording script · expand for backup answers
If asked: What is the star schema?

route_options is the fact table — each row is one commute option with measurable values: duration, fare, walk distance. weather_forecast, bus_arrivals, and train_alerts are dimension tables that join to routes. route_options is the centre of the star; dimensions radiate outward.

If asked: What is model drift? How do you detect it?

Three types from Day 4: Data drift — input feature distributions change (e.g. my commute hour shifts from 8 AM to 10 AM). Concept drift — the relationship between features and target changes (e.g. a new MRT line opens; same hour now predicts shorter journey). Training-serving skew — a feature is computed differently in training vs. production (e.g. is_rainy encoded as True/False in training but 1/0 in the pipeline). My primary signal is the 7-day MAE threshold — rising MAE is a symptom of all three regardless of root cause. Statistical tests like PSI or KS-test per feature can be added later.

If asked: Why Random Forest and not Linear Regression?

Random Forest handles non-linear interactions — rush hour combined with rain has a different effect than either alone. Linear Regression would require manual feature engineering for that. RF also handles small datasets gracefully, which matters because I'm bootstrapping with synthetic data until enough real runs accumulate.

If asked: Why batch processing and not real-time streaming (Kafka)?

Bus arrival data refreshes every 2 minutes. Weather updates every 2 hours. Airflow runs the batch DAG every 30 minutes; the Docker scheduler adaptively polls for live snapshots — nothing changes faster than that. Kafka would add infrastructure complexity (brokers, consumer groups, partitions) with no benefit at this data velocity.

If asked: What would you do differently if the system scaled to many users?

Replace FastAPI with NVIDIA Triton Inference Server for high-throughput batched ML inference; replace pandas with cuDF from the RAPIDS ecosystem for GPU-accelerated data processing; add Metabase as a BI dashboard for non-technical staff. But the Day 5 Technology Selection Exercise maps "low-traffic prediction API" to FastAPI and "100MB daily data" to pandas — which is exactly what I built. Knowing when not to use a tool is as important as knowing how to use one.

If asked: How does the geocoder handle varied address formats?

Progressive fallback: (1) extract a 6-digit postal code with re.findall(r'\b\d{6}\b', address) — postal codes geocode with near-perfect accuracy on OneMap; (2) full address string; (3) address with ", Singapore" stripped; (4) first comma-delimited token. Word boundary \b prevents matching 7-digit phone numbers.

If asked: How does the stale event fix work?

After ingest, _purge_stale_events(con, event_id) deletes all future calendar_events rows whose primary key is not the current active event_id, along with their route_options and route_legs children. Cascading delete order: route_legs first (FK child), then route_options, then calendar_events. Without this, a rescheduled event leaves both the old and new start_time in the DB and transform picks whichever appears first.
