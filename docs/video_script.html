Business Problem & Target Audience

Say This

Singapore has one of the world's most complex public transport networks — over 5,000 bus stops, 6 MRT lines, and weather that changes every 2 hours. Commuters make routing decisions with incomplete information.

The target audience is Singapore-based office workers who use public transport daily. They have a Google Calendar event, they know the destination, but they don't know which route to take right now — factoring in train disruptions and whether it's going to rain during their walk.

The business output is one sentence: "Leave by 09:23 AM — take Bus 65 then East-West Line. Rain expected, avoid the long walk." That's what the system produces.

From a data engineering perspective, this is a real-time enrichment pipeline: four different data sources get ingested, joined in DuckDB, ranked with SQL window functions, and served via both a web dashboard and a REST API.

Say This

The pipeline uses three data sources. Two require API keys.

First, LTA DataMall — this is Singapore's Land Transport Authority developer portal. I registered for free access and received an Account Key. This gives me two endpoints: real-time bus arrival times by bus stop code, and active MRT service alerts with severity levels.

Second, OneMap Singapore — this is the government's official geocoding and routing API. I registered using my email and get a JWT token that expires every 3 days. OneMap gives me two things: geocoding — turning an address like "NUS Shaw Foundation Alumni House" into GPS coordinates — and public transport routing, which returns up to 3 route options with total travel time, walking distance, number of transfers, and estimated fare.

Third, data.gov.sg — this is a completely open API, no registration needed. It gives me the 2-hour weather forecast broken down by area across Singapore.

All three API keys are stored in a file called config.py which is listed in .gitignore. It never goes to GitHub. You can see the template in config_example.py — only placeholders, never real keys.

Talking Point

If asked: "Why three APIs instead of one?" — Each API has a different owner, different auth model, and different update frequency. Combining them is the engineering problem.

Project Structure & Design Decisions
4:30 – 6:00
Screen Action

  • Click through: .gitignore, requirements.txt, config_example.py
  • Open docs/ARCHITECTURE.md briefly
Say This

The folder structure follows a standard data engineering layout. scripts/ holds all pipeline code. data/raw/ stores raw API responses as Parquet files — date-partitioned so every API call is auditable. data/processed/ holds cleaned data. db/ holds the DuckDB database file, which is a single binary file — no server needed. dags/ will hold the Airflow orchestration DAG.

I made three key architectural decisions early. First: DuckDB instead of PostgreSQL. DuckDB is embedded — zero infrastructure to set up, perfect for a local pipeline. It handles analytical SQL including window functions, which I need for route ranking.

Second: idempotency. Every insert uses INSERT OR REPLACE. If I run the pipeline twice, the data doesn't double-up. This is essential for a scheduler running every 10 minutes.

Third: a raw Parquet zone before DuckDB. I save every API response to a Parquet file first. If my SQL transform has a bug later, I can fix it and re-run the transformation from the saved files without hitting the APIs again.

Say This — Code Walkthrough

schema.py is the first script in the build order because everything else depends on the database existing first. It runs once — or safely any number of times — and creates 7 tables plus one SQL view.

The path to the database uses Python's pathlib.Path — I calculate the db folder relative to this script's location, so the code works regardless of which directory you run it from. Before connecting, I call mkdir(parents=True, exist_ok=True) — that creates the db/ folder if it doesn't exist yet, without crashing if it already does.

Every table uses CREATE TABLE IF NOT EXISTS — so re-running schema.py on a database that already has data won't wipe anything. It's a no-op if the table is already there.

I use TIMESTAMPTZ — timestamp with time zone — for event start times and arrival times because Singapore commuters care about real clock time in the SGT +08:00 timezone. I use plain TIMESTAMP for internal tracking columns like fetched_at and ran_at — those are recorded when the pipeline runs and timezone ambiguity doesn't matter for monitoring.

The script uses Python's logging module, not print statements. In production pipelines, print statements don't integrate with monitoring tools. Every log line gets a timestamp automatically, which makes it easier to debug timing issues.

The connection is wrapped in a try/finally block that always calls con.close(). DuckDB only allows one writer at a time — if this script crashed without closing the connection, the next script would fail with a lock error. finally guarantees the lock is released.

Say This — After Running

After running the script, three log lines confirm: 7 tables created, the view created, and the database path. DuckDB created the commute.duckdb file in the db/ folder — you can see it in the file explorer. It's 192 kilobytes right now because there's no data yet, just the schema.

The view — v_enriched_routes — is worth explaining. It's a SQL view, not a table. It doesn't store data itself; it's a saved query. Every time I run a SELECT on it, it executes the join logic live against the four source tables. The view uses a window function — ROW_NUMBER() OVER (PARTITION BY event_id ORDER BY duration) — to rank route options per event, with the fastest route getting rank 1. It also applies weather and disruption penalties: if it's rainy and the walk is over 400 metres, that option gets pushed down the ranking.

If Asked: Why a View and Not a Table?

A view always reflects the latest data — every time ingest.py adds new bus arrivals or weather, the view result changes automatically. A table would need to be re-populated manually. I'll only write to the recommendations table after the view picks the rank-1 route.

DuckDB Schema — 7 Tables Explained
11:00 – 12:30
Screen Action
  • Stay in scripts/schema.py, scroll through the TABLES list
  • Or open docs/ARCHITECTURE.md if it has the schema laid out cleanly
Say This

The schema has 7 tables, each capturing a different data source or pipeline output.

calendar_events — the input. Each event has an ID, title, start time, raw destination text, and geocoded GPS coordinates. This is populated first, before any routing can happen.

route_options — three rows per event from OneMap: duration, walking distance, number of transfers, and fare. The foreign key to calendar_events enforces referential integrity.

weather_forecast — one row per area per forecast window, composite primary key on area and valid_start time. The is_rainy boolean is computed at ingest time so the view doesn't need string matching.

bus_arrivals — near-real-time, composite primary key on stop code, service number, and fetch timestamp. This lets me keep a short history of arrival predictions.

train_alerts — active MRT disruptions with severity levels. The view filters for HEAVY severity alerts from the last 30 minutes.

recommendations — the output table. One row per event, written by transform.py after the view picks the best route.

pipeline_runs — monitoring. Every execution logs its source, how many rows it upserted, duration in milliseconds, and whether it succeeded. This is what the FastAPI status endpoint reads.

transform.py — Producing the Recommendation
11:00 – 13:00
Screen Action
  • Open scripts/transform.py in VS Code
  • Switch to terminal, run: python scripts/transform.py
  • Show the full output — event, leave-latest, leave-now, step-by-step legs, weather, alerts
Say This

transform.py is the third script in the pipeline. It reads from the v_enriched_routes SQL view and produces one output row per calendar event in the recommendations table. It only processes the single next upcoming event — ordered by start time, limit 1 — so you never see stale past events.

The view has already done the heavy lifting — joining routes, weather, and train alerts, computing the leave-by time using a 10-minute buffer, and ranking routes with a SQL window function. transform.py picks rank 1 and formats the result.

The output has two departure times. "Leave latest" is the latest you can leave and still arrive 10 minutes early. "Leave now" tells you what happens if you left at this exact moment — always shown, so you can see exactly how early or late you'd arrive.

The step-by-step legs come from the route_legs table — walk to bus stop, Bus 96, East West Line, walk to destination. Each leg shows the mode icon, duration, and from→to stop names.

Rain and disruption warnings are applied per leg. If it's raining and any walk segment is over 400 metres, that specific walk is flagged. If there's a HEAVY MRT alert from the last 30 minutes, it appears with the affected line.

There's also a walk alternative section. If the destination is under 5 km from my current location and it's not raining, the pipeline checks whether walking is viable. It shows estimated time, steps, calories, and explains which heart rate zone the walk would target — Zone 1 for light recovery or Zone 2 for aerobic base building. If I've configured Garmin or Whoop credentials, it also shows today's step count progress and my recovery score so I can make a more informed decision.

If Asked: How does it know where you are?

The routing origin is detected automatically. If HOME_ADDRESS is set in config.py, it's geocoded via OneMap for address-level precision. Otherwise, the pipeline calls ip-api.com — a free public IP geolocation API — to get your approximate location. In Singapore's compact geography, city-level accuracy is close enough for route ranking. If that fails (e.g. VPN), it falls back to a central Singapore default.

If Asked: Why Garmin and Whoop specifically?

Garmin has a usable Python library (garminconnect). Whoop has a developer API with personal access tokens. Both are optional — the walk suggestion appears without them. Future integrations could include Samsung Health or Apple Health via their respective export APIs.
